# Signature Security Landscape: Common Vulnerability Patterns Across Four Signing Systems

> Author: Mingyang Fan (@SymmaTe)
> Published: 2026-03

---

Signatures are the core authorization mechanism in blockchain security — they prove that "this operation was permitted." But across audits, I keep seeing the same pattern: **the problem almost never lies in the signing algorithm itself, but in how the algorithm is used.**

Missing context in hash construction, bypassable verification logic, signatures reused in the wrong context, the gap between mathematical validity and business validity — every signing system has its own traps.

This article walks through each signing type: what it is, how it works, and the most common vulnerability patterns found in audits.

---

## I. ECDSA + EIP-712

### How It Works

ECDSA (Elliptic Curve Digital Signature Algorithm) is Ethereum's foundational signing algorithm. The signing flow:

```
private key + hash → signature (r, s, v)
ecrecover(hash, signature) → public key / address
```

ECDSA itself is completely indifferent to the contents of the hash — it only handles the math. Security relies entirely on **whether the hash binds sufficient context**.

EIP-712 is the hash construction standard. It defines a structured data serialization format and enforces a domain separator inside every hash:

```
domain separator = hash({
    chainId,           ← binds the chain
    verifyingContract, ← binds the contract
    name, version      ← binds the protocol version
})

final hash = hash("\x19\x01" + domain_separator + struct_hash)
```

ERC1271 is the signature verification standard for smart contract wallets. It defines the `isValidSignature(bytes32 hash, bytes sig)` interface, allowing smart contracts to verify signatures just like EOAs.

### Common Vulnerabilities

**1. ERC1271 Signature Replay: Missing Context Binding in Hash**

`isValidSignature` implementations that verify EOA signatures typically only check "does the recovered address match the owner" — they don't validate how the hash was constructed. If the hash is missing these bindings:

| Missing Binding | Consequence |
|--|--|
| chainId | Cross-chain replay: same signature is valid on other chains |
| verifyingContract | Cross-account replay: multiple smart wallets controlled by the same EOA can replay each other's signatures |
| nonce / usage flag | Historical replay: a used signature can be submitted again |

### Case Study: SSO Account OIDC Recovery — ERC1271 Signature Replay

> Report: [SSO Account OIDC Recovery Solidity Audit · May 2025](https://solodit.cyfrin.io/issues/potential-signature-replay-attack-in-erc1271handler-openzeppelin-none-sso-account-oidc-recovery-solidity-audit-markdown)

```
Scenario 1: Historical replay
User previously signed authorization for operation A (hash_A + sig_A)
    ↓
Attacker finds that scenario B also accepts the same hash_A
    ↓
isValidSignature(hash_A, sig_A) returns valid → scenario B executes

Scenario 2: Cross-chain replay (hash lacks chainId)
User signs on Chain A
    ↓
Attacker submits the same hash + sig to Chain B
isValidSignature returns valid on Chain B as well

Scenario 3: Cross-account replay (hash lacks verifyingContract)
Same EOA controls Account_1 and Account_2
User signs an operation on Account_1
    ↓
Attacker submits hash + sig to Account_2 → passes validation
```

Fix: Use ERC7739's defensive rehashing scheme, wrapping the app-layer hash in an additional EIP-712 layer to enforce binding to chainId and verifyingContract. Solady's `ERC1271.sol` provides a standard implementation.

---

**2. Typehash Typo: All Signatures Permanently Broken**

```solidity
// Bug: typo causes typehash to differ from what wallets compute
bytes32 constant MESSAGE_TYPEHASH =
    keccak256("SnowmanClaim(addres receiver, uint256 amount)");
//                          ↑ missing 's'

// Wallets generate signatures using the correct spelling
// → typehash mismatch → nobody can claim
```

The result is a complete functional DoS — all legitimate signatures fail verification. This class of bug is easy to miss in tests because the test code uses the same (broken) typehash as the contract. It only surfaces when interacting with a standard wallet.

### Case Study: Snowman Merkle Airdrop — MESSAGE_TYPEHASH Typo

> Report: [Snowman Merkle Airdrop Audit · Mar 2026 (personal audit)](https://github.com/SymmaTe/my-audit-reports/blob/main/2026-03-08-MerkleAirdrop-audit.pdf)

```
Contract: keccak256("SnowmanClaim(addres receiver, uint256 amount)")
                                   ↑ missing 's'
Wallet:   keccak256("SnowmanClaim(address receiver, uint256 amount)")

The two typehashes differ
    ↓
Every signature generated by a standard wallet fails verification
No user can claim their Snowman NFT
```

**3. Signature Flag Manipulation: Verification Path Hijacked**

The signature itself is valid, but a bypass exists in the verification logic. For example, a bitmask might indicate "whether a verification module is enabled." An attacker appends a self-crafted flag to their signature, causing the verifier to include it in the hash computation and thereby control which verification path executes.

Root cause: the signature only binds "who signed it," not "what configuration was used to verify it."

### Case Study: Sequence — Bitmask Flag Manipulation Bypasses Checkpointer

> Report: [Sequence · Code4rena](https://solodit.cyfrin.io/issues/h-01-chained-signature-with-checkpoint-usage-disabled-can-bypass-all-checkpointer-validation-code4rena-sequence-sequence-git)

```
Precondition: the attacker is a legitimate signer who wants to bypass
              the additional restrictions imposed by checkpointer.

Normal flow:
  signature = sign(hash) + checkpointerEnabled flag
  verifier reads flag → enables checkpointer → performs additional checks

Attack:
  Attacker constructs flag = disabled, appends it to the signature
    ↓
  Verifier includes attacker's flag in hash computation
  Attacker signs the hash containing flag=disabled with their own key → signature is valid
    ↓
  Checkpointer is disabled → all checkpoint validation bypassed
```

### Audit Checklist

- In the `isValidSignature` implementation, does the hash include chainId and verifyingContract?
- Does the TYPE_HASH string contain any typos?
- Are there user-controllable configuration parameters in the verification path that are not covered by the signature?
- Is an audited standard library (OpenZeppelin, Solady) being used?

## II. EIP-2612 Permit

### How It Works

EIP-2612 extends ERC20 to allow signatures to replace `approve`, enabling gasless token authorization:

```
User signs: permit(owner, spender, value, deadline, v, r, s)
Anyone can submit this signature on-chain → equivalent to owner calling approve(spender, value)
```

Key property: **the signature can be submitted by anyone** — this is intentional, allowing relayers to pay gas on behalf of users.

### Common Vulnerabilities

**1. permit Call Not Wrapped in try/catch: Griefing DoS**

Once a permit signature enters the mempool, anyone can front-run its submission. If a contract bundles `permit` with subsequent operations in the same transaction and doesn't tolerate `permit` failure, an attacker can copy the signature parameters and submit them first — causing the user's transaction to revert indefinitely because the signature has already been consumed.

### Case Study: LI.FI — Front-Run permit Submission Causes User Transaction Failure

> Report: [LI.FI · Dec 2024](https://solodit.cyfrin.io/issues/griefing-attack-possible-by-frontrunning-the-calldiamondwitheip2612signature-function-call-cantina-none-lifi-pdf)

```
User signs permit → submits callDiamondWithEIP2612Signature → enters mempool
    ↓
Attacker monitors mempool, copies permit parameters, calls ERC20.permit() directly
Attacker's tx is included first: allowance set, signature marked as used
    ↓
User's tx is included:
  ERC20.permit() reverts — signature already used
  Entire transaction fails
```

The attacker gains nothing (the allowance is set for the correct spender), but can continuously block the user from executing transactions.

Correct pattern:

```solidity
// ✅ permit failure is fine — the goal is sufficient allowance, not a successful permit call
try IERC20Permit(token).permit(owner, spender, amount, deadline, v, r, s) {
} catch {}

if (IERC20(token).allowance(owner, spender) < amount) revert InsufficientAllowance();
```

Whether the user or the attacker submitted the permit, the result is the same — allowance is correctly set. A failed permit call doesn't mean the allowance is insufficient.

---

**2. Permit Signature Phishing**

Permit signatures don't trigger an on-chain transaction, so users signing in MetaMask and similar wallets see no gas prompt and tend to be less cautious. A malicious DApp can trick a user into signing a permit targeting a malicious contract, then immediately call `transferFrom` to drain the assets.

---

**3. Deadline Set Too Far in the Future**

Permit signatures include a deadline field. If the deadline is set to the distant future (or even `type(uint256).max`), a leaked signature can be exploited for an extended period.

### Audit Checklist

- Is every `permit` call wrapped in try/catch?
- After try/catch, is the allowance explicitly checked rather than assuming permit succeeded?
- Does the frontend clearly show the user what they are authorizing and for how much?
- Is the deadline set to a reasonable value?

## III. BLS Aggregate Signatures

### How It Works

BLS (Boneh-Lynn-Shacham) signatures have one defining property: **linear additivity**. Multiple signatures can be merged into a single aggregate signature, verified in a single operation, drastically reducing on-chain verification costs. Commonly used for collective validator signatures in PoS consensus.

```
Aggregation:
  sig_A = BLS.sign(privKey_A, msg)
  sig_B = BLS.sign(privKey_B, msg)
  sig_agg = sig_A + sig_B  ← elliptic curve point addition

Verification:
  BLS.verify(pubKey_A + pubKey_B, msg, sig_agg) → true
```

### Common Vulnerabilities

**1. Point at Infinity: Forging Voting Weight with a Blank Entry**

BLS curves (such as BN254) include a special element called the "point at infinity," which is the identity element of the elliptic curve group:

```
Any point P + point at infinity = P
```

If the verification logic doesn't reject the point at infinity as a valid public key, an attacker can register a "transparent key" (pubKey = point at infinity). It doesn't affect the aggregate signature mathematically, but it is counted in voting weight calculations — giving the attacker weight at zero cost.

### Case Study: Symbiotic Relay — Point at Infinity Public Key Forges Consensus Weight

> Report: [Symbiotic Relay · Sherlock Jul 2025](https://solodit.cyfrin.io/issues/m-7-a-malicious-operator-will-control-consensus-without-risking-stake-stake-exit-lag-exploit-sherlock-symbiotic-relay-git)

```
Attacker registers a validator with pubKey = point at infinity
    ↓
Aggregate verification:
  pubKey_agg = legitimate validator set + point at infinity
             = legitimate validator set (mathematically unchanged)
  sig_agg verification passes ✓
    ↓
But in voting weight accounting: attacker's transparent key is counted
    ↓
Attacker gains voting weight with zero stake → may influence consensus outcome
```

Fix: Explicitly reject the point at infinity as a valid public key in both circuit and verification logic.

---

**2. Rogue Key Attack**

BLS aggregates public keys via elliptic curve point addition: `pubKey_agg = pubKey_A + pubKey_B + ...`.

An attacker registers a public key claiming it is `pubKey_evil - pubKey_victim`. The aggregate result:

```
pubKey_agg = pubKey_victim + (pubKey_evil - pubKey_victim) = pubKey_evil
```

The aggregated public key is now entirely under the attacker's control. The attacker can produce a valid aggregate signature for any message unilaterally, without any cooperation from the victim.

Fix: Use Proof of Possession — require every validator to provide a signature over their own public key at registration time, proving they hold the corresponding private key.

### Audit Checklist

- Does the verification logic reject the point at infinity as a public key or signature?
- Is Proof of Possession implemented to prevent rogue key attacks?
- Is participant identity fully verified in aggregate signatures, not just mathematical validity?

## IV. Schnorr Signatures (Bitcoin Taproot)

### How It Works

Bitcoin Taproot (activated in 2021) replaced the underlying signing algorithm from ECDSA to Schnorr. Schnorr's core advantage is **linear additivity** (similar to BLS), enabling multi-signature aggregation (MuSig).

Bitcoin signatures are computed over a **sighash**, not the raw transaction data. The sighash computation rules differ by transaction format:

| Format | Does sighash commit to prevOut data? |
|--|--|
| Legacy (P2PKH, etc.) | No |
| SegWit v0 (P2WPKH) | Yes (amount only) |
| Taproot / SegWit v1 (BIP341) | Yes (scriptPubKey + amount) |

Taproot requires the sighash to commit to each input's previous output (prevOut) — its `scriptPubKey` and `amount` — to prevent hardware wallets from being tricked into signing transactions without knowing the input values.

### Common Vulnerabilities

**1. Incorrect sighash Input Data During Legacy → Taproot Migration**

Legacy sighash computation doesn't depend on prevOut data, so historical code commonly passes placeholder values via `NewCannedPrevOutputFetcher([]byte{}, 0)`. After migrating to Taproot, the sighash rules change fundamentally — the real `scriptPubKey` and `amount` for each UTXO input must be provided. Reusing old code compiles without errors but produces a sighash that doesn't match what the Bitcoin network computes, causing signatures to fail permanently.

### Case Study: ZetaChain — Empty prevOut in Bitcoin Observer Causes All Withdrawals to Be Rejected

> Report: [ZetaChain Cross-Chain · Sherlock May 2025](https://solodit.cyfrin.io/issues/m-38-bitcoin-observers-signed-transactions-will-be-rejected-due-to-invalid-sighashes-sherlock-zetachain-cross-chain-git)

```go
// Old code (Legacy): fine, Legacy sighash doesn't depend on prevOut
sigHashes := txscript.NewTxSigHashes(tx,
    txscript.NewCannedPrevOutputFetcher([]byte{}, 0))

// New code (Taproot): copy-pasted directly → compiles, semantic error
sigHashes := txscript.NewTxSigHashes(tx,
    txscript.NewCannedPrevOutputFetcher([]byte{}, 0))  // ← empty script + 0 amount
```

```
Observer (ZetaChain's off-chain listener node) computes sighash_wrong using empty prevOut → signs
    ↓
Broadcasts to the Bitcoin network
    ↓
Bitcoin nodes fetch real UTXO data from the chain
Recompute sighash_correct (using real scriptPubKey + real amount)
    ↓
sighash_correct ≠ sighash_wrong → signature verification fails
All withdrawal transactions rejected by the network
```

This class of bug is extremely hard to catch in local tests — mock data skips sighash validation. It only surfaces when broadcasting to the real network.

### Audit Checklist

- For Bitcoin transaction signing code, confirm whether it targets Legacy or Taproot format
- Search for uses of `NewCannedPrevOutputFetcher` — its presence in production code is almost always a bug
- Are there integration tests against the real Bitcoin network?

## Unified Perspective

Four signing systems, four categories of problems — but fundamentally all different expressions of the same issue:

| Signing System | Core Trap | Root Cause |
|--|--|--|
| ECDSA + EIP-712 | Incomplete hash context / typehash typo / verification bypass | Signature doesn't bind enough context |
| EIP-2612 Permit | permit call not wrapped in try/catch | Confusing "operation succeeded" with "goal achieved" |
| BLS Aggregate | Point at infinity forges weight / rogue key attack | Mathematical validity ≠ business validity |
| Schnorr / Bitcoin | sighash input uses old-format placeholder | Format migration without updating underlying assumptions |

The fundamental question in signature security isn't "is the algorithm strong enough" — it's:

> **Is the context bound by the signature complete? Are the verification boundaries consistent? Have the underlying assumptions been updated alongside format migrations?**

## Quick Audit Reference

**ECDSA + EIP-712:**
- Does the domain separator include chainId and verifyingContract?
- Does the typehash string contain any typos?
- Are there user-controllable verification config parameters not covered by the signature?
- Does the ERC1271 isValidSignature implementation use ERC7739?

**EIP-2612 Permit:**
- Is every permit call inside a try/catch?
- After try/catch, is allowance checked rather than assuming permit succeeded?

**BLS Aggregate Signatures:**
- Is the point at infinity rejected as a valid public key?
- Is Proof of Possession implemented?

**Schnorr / Bitcoin:**
- Is `NewCannedPrevOutputFetcher` (a test placeholder tool) present in production code?
- Does the sighash computation receive real prevOut data?
- Are there integration tests against the real network?

---

Signature-related code is often the easiest to skip in an audit — `ecrecover` returned an address, `verify` returned true, `permit` is just one line — everything looks fine. But the vulnerability is never in those calls themselves. It's in the surrounding context: what the hash actually binds, under what scenarios the signature is valid, whether bypasses exist in the verification logic, and whether old assumptions still hold after a format upgrade.

**When auditing signature code, instead of checking whether the cryptographic call exists, ask: what does this signature precisely authorize, and can it be replayed or abused in the current context?**
